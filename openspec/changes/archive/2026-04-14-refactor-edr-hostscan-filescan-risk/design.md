## Context

本次变更需要在不丢失既有采集能力的前提下，重构 EDR CLI 的命令入口与参数体系。当前实现中扫描与分析入口分散（`account scan`、`port-scan`、`file scan`、`risk analyze` 等），并且各命令分别维护输出参数（`--excel`、`--json`），导致组合使用复杂、帮助信息冗余、测试覆盖分散。用户已确认目标形态为：`hostscan` 与 `filescan` 两个主扫描模块，`-r` 为全局风险分析开关，`-o` 通过输出路径后缀统一输出格式。

约束条件：
- `--custom` 与 `--all` 必须互斥。
- `hostscan/filescan` 未指定模块时默认 `all`。
- 多模块/all 参数仅取交集。
- 去重规则仅为“记录完全一致去重”。
- 错误提示统一中文。

## Goals / Non-Goals

**Goals:**
- 建立统一 CLI 入口：`edr hostscan`、`edr filescan`，并保留 `-r`、`-o` 全局行为。
- 把风险分析改造成可独立执行、也可串联扫描执行的全局能力。
- 实现基于输出路径后缀的统一结果输出（JSON/CSV/XLSX）。
- 明确参数冲突规则与中文错误提示，确保自动化脚本可预测。
- 让改造后主流程只暴露两类扫描模块（hostscan/filescan）。
- 保证真实数据场景下 `filescan/hostscan + -r` 的稳定性（异常样本不崩溃、空结果可返回）。
- 在不降低风险结论准确性的前提下，减少云分析链路中的无效请求。

**Non-Goals:**
- 不改动各底层采集器的核心采集算法（如账户枚举、端口探测、YARA 匹配内部逻辑）。
- 不在本次变更中引入新的云分析 provider 或新的风险评分模型。
- 不保证旧命令路径长期兼容；本次以新入口为主。

## Decisions

- **决策 1：采用“全局参数预解析 + 模块参数二次解析”架构**
  - 方案：先解析全局 `-r/--riskanalyze`、`-o`、`-h`，再把余下参数分发给 `hostscan/filescan` 专属解析器。
  - 原因：避免 `-r/-o` 在不同子命令重复定义，减少参数冲突。
  - 备选：各子命令重复声明全局参数；缺点是实现重复且行为易漂移。

- **决策 2：`filescan` 内区分“Web聚合模式”和“本地文件模式”**
  - 方案：
    - `--custom/--all`（或默认 all）走 `site/framework/jarpackage` 聚合扫描。
    - 显式提供本地文件扫描参数时走本地扫描流程（默认 smart）。
    - 两类模式互斥，避免语义混叠。
  - 原因：满足“site/framework/jarpackage 与 full/path/smart 不混用”的要求，同时保留 `filescan --... -r` 本地串联分析场景。
  - 备选：只保留一种模式；缺点是无法覆盖用户已确认的两类调用需求。

- **决策 3：风险分析模式参数改名为 `--risk-mode`（保留 `smart` 默认）**
  - 方案：将风险分析模式参数与扫描模式参数解耦，避免 `--mode` 歧义。
  - 原因：用户已指出 mode 存在重名冲突，允许自定义区分。
  - 备选：继续复用 `--mode`；缺点是 `filescan` 下无法区分扫描模式与分析模式。

- **决策 4：统一输出采用 `-o <path>` 后缀识别格式，未传 `-o` 自动导出 Excel**
  - 方案：根据 `-o` 路径后缀识别 `.json/.csv/.xlsx`；使用 `-o` 时必须指定路径；未传 `-o` 时自动在当前目录导出 `result*.xlsx`（按 `result.xlsx`、`result1.xlsx`、`resultN.xlsx` 递增）。
  - 原因：符合“路径即格式声明”与“默认落盘可追溯”的要求，减少参数数量并避免遗漏输出文件。
  - 备选：未传 `-o` 时 stdout JSON；缺点是和当前默认落盘策略不一致，且批量执行不便留存结果。

- **决策 10：filescan 的 path 模式仅保留 `--scan-mode path <path>`**
  - 方案：移除 `--scan-path`，统一为位置参数写法 `--scan-mode path <path>`。
  - 原因：减少等价参数写法，降低帮助信息与参数校验复杂度。
  - 备选：同时保留 `--scan-path` 与位置参数；缺点是提示冗余、用户输入分歧更大。

- **决策 5：多模块结果按“规范化后完全相等”去重**
  - 方案：将记录做稳定键序列化后比较，完全相等才去重。
  - 原因：用户明确要求仅“完全一样”才去重，不做包含关系合并。
  - 备选：字段包含关系合并；缺点是行为不可解释、实现复杂且已被否决。

- **决策 6：聚合参数仅使用子模块参数交集**
  - 方案：为 `hostscan/filescan` 的多模块/all 预先定义参数交集白名单；单模块使用对应模块完整参数集。
  - 原因：符合需求并降低“传了某模块不支持参数”造成的歧义。
  - 备选：参数并集；缺点是需要按模块做条件校验，错误体验更差。

- **决策 7：串联风险分析在零记录场景返回空结果而非失败**
  - 方案：`hostscan/filescan + -r` 在扫描记录为空时直接返回 `[]`，退出码保持 0。
  - 原因：空数据是真实环境的正常情况（例如某模块当前主机无数据），不应被视为执行失败。
  - 备选：保持“no scan records provided”错误；缺点是自动化任务需要额外容错，且语义不一致。

- **决策 8：本地二进制解析采用 panic-safe 兜底**
  - 方案：对 PE 导入表读取（`ImportedSymbols/ImportedLibraries`）增加 `recover` 防护，将崩溃转为可恢复错误并跳过该字段。
  - 原因：真实环境存在损坏或非标准 PE，单样本异常不应影响整批扫描。
  - 备选：维持直接调用；缺点是会触发进程级 panic，影响可用性。

- **决策 9：云 provider 准入与熔断**
  - 方案：初始化时跳过无 API key provider；运行时若 provider 返回鉴权/密钥类错误（401/403/unauthorized/api key）则在本次执行中禁用该 provider。
  - 原因：避免重复无效调用，降低 `smart` 场景下大量目标时的无效耗时，同时不改变已有风险评分模型。
  - 备选：维持每条记录都尝试；缺点是无意义重试拖慢整体分析。

## Risks / Trade-offs

- **[兼容性破坏] 旧命令脚本失效** → 通过中文错误提示与升级说明引导迁移到 `hostscan/filescan`。
- **[参数复杂度上升] filescan 双模式可能让用户困惑** → 在 `-h` 中分组展示示例，并为互斥组合给出明确报错。
- **[输出格式扩展风险] CSV/XLSX 序列化字段一致性风险** → 复用统一中间结果模型，再做格式化导出，配套回归测试。
- **[交集参数过窄] 多模块可用参数减少** → 在帮助信息中显式区分“单模块参数全集”与“聚合参数交集”。
- **[provider 可用性下降] 跳过无 key 或熔断后可用云源减少** → 保持多 provider 聚合与 fail-safe 规则，且仅在明确鉴权错误时熔断。
- **[性能优化边界] 降低无效请求不等于云查询整体提速** → 保持策略“先去无效请求，再由配置/配额决定吞吐”，避免牺牲准确性换速度。

## Migration Plan

1. 引入新根路由与全局参数预解析器，完成 `hostscan/filescan/-r/-o` 命令骨架。
2. 接入 hostscan/filescan 聚合执行器与结果去重器，打通不带 `-r` 的扫描输出。
3. 接入 `-r` 串联流程（扫描后只输出分析结果）与独立 `-r -file` 文件分析流程。
4. 接入统一输出器（未传 `-o` 自动导出 `result*.xlsx`；传 `-o` 按后缀导出 JSON/CSV/XLSX）。
5. 更新 `-h` 文案、中文错误提示与相关单元测试/集成测试。
6. 验证改造后外部主流程只保留 `hostscan/filescan` 两个扫描模块。
7. 补充真实数据深度测试，覆盖 `filescan --scan-mode smart -r` 与 `hostscan -r` 的稳定性、空结果与耗时边界。
