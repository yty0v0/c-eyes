## Context

当前 EDR 缺少针对 Web 框架层面的统一信息采集能力，导致资产台账中框架名称、版本、部署目录、服务类型和关联 jar 信息难以统一沉淀。该能力要求在 Windows 与 Linux 端均可运行，通过 CLI 触发执行，但采集过程不能依赖外部命令行命令，仅做信息收集，不进入风险分析。

需求侧明确了过滤参数（`groups`、`hostname`、`ip`、`name`、`version`、`type`、`serverName`）与返回字段契约（主机字段、框架字段、站点字段、目录字段、jar 详情字段）。另外，内外网 IP 需改为“全量数组化采集”，与已有扫描模块在多网卡场景下的表达保持一致。

## Goals / Non-Goals

**Goals:**
- 提供 `web-framework-scan` 跨平台能力，在 Windows/Linux 输出一致字段结构。
- 采集策略采用静态 + 动态结合（配置/文件特征 + 运行态关联）以提升识别覆盖率。
- 保证采集阶段不通过外部命令执行获取信息。
- 支持组合过滤查询并输出稳定结果契约。
- 支持 JSON 与 Excel 两种导出格式，统一 UTF-8 编码。
- 将内外网 IP 以列表字段输出，覆盖多网卡全量地址。

**Non-Goals:**
- 不输出风险评分、漏洞判定、威胁等级与处置建议。
- 不在本变更内引入主动扫描、漏洞利用或网络攻击面探测。
- 不重构其它已上线扫描能力的现有契约，仅复用其可复用组件。

## Decisions

1. 统一领域模型驱动
- 决策：定义统一 `WebFrameRecord`、`JarRecord` 数据模型，由平台适配器产出后汇总。
- 原因：统一过滤、导出、测试逻辑，减少平台分叉。
- 备选方案：平台侧各自输出不同结构后在导出处兼容；被拒绝，因为维护成本高且容易出现字段漂移。

2. 采集采用“静态 + 动态”双通道
- 决策：静态通道解析配置文件、目录特征、部署元文件；动态通道关联运行进程、加载路径与运行参数，最后做去重合并。
- 原因：单一路径会导致识别漏报，双通道更适配复杂部署。
- 备选方案：仅静态配置解析；被拒绝，因为无法覆盖运行时路径漂移和非默认部署。

3. 非命令行采集约束
- 决策：采集仅使用进程内 API、文件系统读取、配置解析、系统信息接口，不调用外部命令。
- 原因：满足约束要求并降低命令注入/环境依赖风险。
- 备选方案：通过 `ps`、`wmic`、`netstat` 等命令辅助采集；被拒绝，违反约束。

4. IP 字段契约数组化
- 决策：输出 `displayIp`（展示用）+ `internalIpList`/`externalIpList`（全量集合），不再依赖单值 `internalIp`/`externalIp`。
- 原因：满足多网卡与多地址场景，避免信息丢失。
- 备选方案：保留单值字段并追加数组字段；被拒绝，避免双轨字段造成下游歧义。

5. 导出层采用 writer 抽象
- 决策：实现统一导出接口，至少提供 JSON Writer 与 Excel Writer，两者共享同一字段映射。
- 原因：保持数据契约一致，便于测试与后续扩展。
- 备选方案：在命令层分支拼装输出；被拒绝，因为格式分支会复制业务逻辑。

## Risks / Trade-offs

- [Risk] 框架识别规则在不同发行版/安装方式下可能漏识别。 -> Mitigation: 维护可扩展特征库并补充多平台样本测试。
- [Risk] 动态关联可能将同主机多个服务错误合并。 -> Mitigation: 采用多键匹配（进程路径、监听端口、配置根目录）并加入冲突回退策略。
- [Risk] `jarList` 规模较大时 Excel 导出体积和内存压力上升。 -> Mitigation: 使用流式写入并限制单次导出条数/提供分页参数。
- [Risk] 历史下游依赖单值 IP 字段会产生兼容风险。 -> Mitigation: 在发布说明中明确字段变更并提供迁移映射示例。

## Migration Plan

1. 增加 `web-framework-scan` 命令入口及参数解析/校验。
2. 建立统一记录模型与过滤管线，接入主机信息补全和 IP 列表聚合。
3. 实现 Windows/Linux 采集适配器，确保不通过外部命令执行采集。
4. 实现 JSON/Excel 导出器并统一字段映射。
5. 增加单元测试与集成测试，验证字段稳定性、格式一致性与边界约束。

回滚策略：
- 若上线后发现兼容性问题，可在命令注册层临时下线 `web-framework-scan` 子命令，不影响既有扫描能力。

## Open Questions

- `type` 与 `serverName` 是否需要固定枚举（如 `java/php/python`、`nginx/apache/tomcat`）还是允许透传扩展值？
- `jarCount` 字段是否最终按字符串输出（兼容需求表）或调整为数值型（便于统计）？
- Excel 中 `jarList` 采用单元格 JSON 字符串，还是拆分为副表结构？
