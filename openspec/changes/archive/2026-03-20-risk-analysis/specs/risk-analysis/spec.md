## ADDED Requirements

### Requirement: Risk input sources and mutual exclusion
The system SHALL support exactly one risk-analysis input source per execution: `-input`, `-file`, `-dir -r`, `-pid`, or `-pname`.

#### Scenario: Reject mixed input sources
- **WHEN** a user provides `-input` together with `-file`
- **THEN** the analyzer returns a validation error that input sources are mutually exclusive

#### Scenario: Require recursive directory mode
- **WHEN** a user provides `-dir` without `-r`
- **THEN** the analyzer returns a validation error and exits without analyzing records

### Requirement: Support JSON and Excel scan record loading
The system SHALL parse scan records from JSON array, NDJSON, and Excel (`.xlsx`) files via `-input`. Unknown fields MUST be tolerated.

#### Scenario: Parse Excel generated by scanner
- **WHEN** a user provides an Excel result generated by this project
- **THEN** the loader converts rows into normalized scan records and proceeds with analysis

### Requirement: Support analysis modes and parameters
The system SHALL support `local_only`, `cloud_only`, and `hybrid` analysis modes. The `risk_assessment.analysis_mode` field MUST reflect the effective mode.

#### Scenario: Local-only analysis mode
- **WHEN** analysis is executed with mode `local_only`
- **THEN** the output includes populated `local_analysis` and an empty or omitted `cloud_analysis`

### Requirement: Local engine availability and mode fallback policy
The system SHALL enforce strict local behavior when YARA-X is unavailable:
- `local_only`: fail fast with a clear error.
- `hybrid`: emit a warning and fall back to `cloud_only`.

#### Scenario: Local-only without YARA-X
- **WHEN** mode is `local_only` and the local YARA-X engine cannot initialize
- **THEN** the command exits with an explicit error indicating local analysis cannot continue

#### Scenario: Hybrid without YARA-X
- **WHEN** mode is `hybrid` and the local YARA-X engine cannot initialize
- **THEN** the command continues in `cloud_only` mode and logs a warning

### Requirement: Local YARA-X matching
When local analysis is enabled, the system SHALL load YARA-X rules via an embedded library call and evaluate scan targets locally. It MUST set `local_matched` and output a `yara_results` array containing `rule_name`, `namespace`, `tags`, `severity`, and optional `matched_strings`.

#### Scenario: Local rule match detected
- **WHEN** local YARA-X matching identifies one or more rules for a scan target
- **THEN** `local_matched` is `true` and `yara_results` contains all matched rules with their metadata

### Requirement: Cross-host local safety fallback
For path-based local matching, the system MUST avoid cross-host false analysis by enforcing source identity checks:
- If `hostname` exists and differs from current host, skip local matching.
- If file hash is provided and does not match current file at `target_path`, skip local matching.
- If both source hostname and hash are missing, skip local matching.
On skip, `local_analysis.local_fallback=true` and `local_fallback_reason` MUST be set.

#### Scenario: Hostname mismatch fallback
- **WHEN** input hostname indicates a different host than the current machine
- **THEN** local matching is skipped with `local_fallback=true`

#### Scenario: Hash mismatch fallback
- **WHEN** input hash does not match the local file found at the same path
- **THEN** local matching is skipped with a hash-mismatch fallback reason

### Requirement: Process memory optional local analysis
The system SHALL support optional process-memory analysis in risk mode:
- `-process-memory` is only valid with `-pid` or `-pname`.
- process-source analysis SHALL still include standard `process` target records.
- when enabled, additional `process_memory` target records SHALL be generated and analyzed locally.

#### Scenario: Process memory enabled for PID source
- **WHEN** a user runs risk analysis with `-pid 1234 -process-memory`
- **THEN** the analyzer generates both `process` and `process_memory` records for the process

#### Scenario: Process memory used without process source
- **WHEN** a user provides `-process-memory` without `-pid` or `-pname`
- **THEN** the command returns a validation error

### Requirement: Process memory capture fallback behavior
Process memory capture failures SHALL be record-scoped and non-fatal. The analyzer MUST continue and mark local fallback for that memory record with a clear reason.

#### Scenario: Memory capture permission denied
- **WHEN** memory capture fails for a process due to permission restrictions
- **THEN** analysis continues for other records and the failed memory record reports `local_fallback=true` with reason

### Requirement: Chunked local file reading for YARA
The local YARA file path scan SHALL read file content in chunks and support a configurable chunk size via `-yara-read-chunk` (must be > 0).

#### Scenario: Custom YARA chunk size
- **WHEN** a user provides `-yara-read-chunk 8388608`
- **THEN** local file reads use 8 MiB chunks for scan input

### Requirement: Cloud threat intel query
When cloud analysis is enabled, the system SHALL query threat intel using the target hash (prefer `sha256`). It MUST set `cloud_queried` and, on success, populate `malicious_votes`, `total_engines`, `detection_ratio`, `threat_labels`, and optional `cloud_link`.

#### Scenario: Cloud query succeeds
- **WHEN** cloud analysis is enabled and the target has a `sha256` hash
- **THEN** the output includes populated cloud metrics and `cloud_queried` is `true`

### Requirement: Multi-provider cloud integration and config ownership
The cloud analyzer SHALL support these providers in parallel aggregation: `virustotal`, `hybrid_analysis`, `malwarebazaar`, `otx`, `triage`. Provider connection fields (`api_key`, `base_url`, `proxy_url`, `rate_limit`, `timeout`, `cache_ttl`) MUST be managed via cloud config.

#### Scenario: Provider-specific override of global proxy
- **WHEN** top-level `proxy_url` and `providers.triage.proxy_url` are both configured
- **THEN** triage requests use provider-level proxy while other providers use top-level proxy

#### Scenario: Provider fields are config-driven
- **WHEN** cloud provider fields are already defined in cloud config
- **THEN** risk analysis does not require duplicate provider URL/key fields in runtime request parameters

### Requirement: Provider base URL defaults
If `base_url` is not configured, the system SHALL use provider defaults:
- `virustotal`: `https://www.virustotal.com`
- `hybrid_analysis`: `https://hybrid-analysis.com/api/v2`
- `malwarebazaar`: `https://mb-api.abuse.ch/api/v1/`
- `otx`: `https://otx.alienvault.com`
- `triage`: `https://tria.ge/api/v0`

#### Scenario: Use default VirusTotal base URL
- **WHEN** virustotal provider has no `base_url` in config
- **THEN** virustotal requests are sent to `https://www.virustotal.com`

### Requirement: Cloud behavior without API keys
Missing API key for one provider SHALL not fail the entire cloud analysis. The analyzer MUST continue with available providers and aggregate partial results.

#### Scenario: No API key for one provider
- **WHEN** `otx` has no API key but other providers are configured
- **THEN** the analyzer skips `otx` and still returns cloud results from remaining providers

### Requirement: Provider-aware cloud weighting
Cloud score aggregation SHALL apply provider-specific weights and high-confidence floor rules:
- `virustotal`: 1.0
- `hybrid_analysis`: 0.95
- `malwarebazaar`: 1.25
- `otx`: 0.45
- `triage`: 1.05
If malwarebazaar indicates high-confidence malicious hit, final cloud score floor MUST be 95; if triage score is high-risk, floor MUST be 85.

#### Scenario: MalwareBazaar high-confidence hit
- **WHEN** malwarebazaar returns a high-confidence malicious result
- **THEN** aggregated cloud score is at least 95

### Requirement: Weighted risk scoring and levels
The system SHALL compute a final `risk_score` in the range 0-100 using configurable weights for local and cloud results. It MUST map scores to risk levels using: 0-20 无风险, 21-50 低风险, 51-80 中风险, 81-100 高风险.

#### Scenario: Hybrid scoring produces high risk
- **WHEN** local and cloud results produce a weighted score of 88
- **THEN** `risk_level` is `高风险` and `risk_score` is 88

### Requirement: Target type coverage
Risk analysis target types SHALL include `file`, `process`, and `process_memory`.

#### Scenario: Process source without memory mode
- **WHEN** process source analysis runs without `-process-memory`
- **THEN** output records use `target_type=process` only

### Requirement: JSON output schema
The system SHALL output JSON records that include target metadata (`scan_id`, `timestamp`, `target_type`, `target_path`, `pid`, `file_size`, `hashes`) and analysis sections (`risk_assessment`, `local_analysis`, `cloud_analysis`).

#### Scenario: JSON output includes required fields
- **WHEN** analysis completes for a scan record
- **THEN** the JSON output includes all required target metadata and analysis sections

### Requirement: Excel output
The system SHALL support exporting analysis results to Excel, with one row per scan target and columns for the required metadata, `risk_score`, `risk_level`, and mode-specific fields.

#### Scenario: Excel output requested
- **WHEN** the output format parameter is set to Excel
- **THEN** an Excel file is produced containing the analysis results for each scan target

### Requirement: Proxy scope isolation
Proxy settings SHALL apply only to cloud HTTP calls and MUST NOT affect local YARA analysis.

#### Scenario: Local-only analysis with proxy configured
- **WHEN** mode is `local_only` and cloud proxy is configured
- **THEN** local YARA analysis behavior is unchanged and does not depend on proxy connectivity

### Requirement: Cloud config file discovery order
Cloud configuration MUST support automatic path discovery in this order:
1) `EDR_CLOUD_CONFIG`
2) `<exe-dir>/edr-cloud.json`
3) `./edr-cloud.json`
4) `~/.edr/cloud.json`

#### Scenario: Read config beside executable
- **WHEN** `edr-cloud.json` exists in the same directory as the executable
- **THEN** cloud analysis loads API keys/settings from that file without requiring CLI key flags

#### Scenario: Root template used for packaging handoff
- **WHEN** a maintainer keeps `edr-cloud.json` in project root but does not bundle it into `dist`
- **THEN** users can place that config next to the executable to enable cloud analysis without recompiling

### Requirement: Cloud aggregation output identity
In multi-platform cloud mode, output MUST explicitly describe aggregation identity:
- `cloud_analysis.cloud_provider` MUST be `multi`
- `cloud_analysis.cloud_providers` MUST list providers that successfully contributed results

#### Scenario: Partial provider success
- **WHEN** only `virustotal` and `otx` return successful cloud results
- **THEN** output contains `cloud_provider=multi` and `cloud_providers=["virustotal","otx"]`

### Requirement: Deprecated single-provider selector behavior
Legacy single-provider selector inputs (`-cloud-provider`, `EDR_CLOUD_PROVIDER`) SHALL NOT change execution mode from multi-provider aggregation. The system MAY emit a deprecation warning.

#### Scenario: Deprecated cloud-provider flag is passed
- **WHEN** a user executes risk analysis with `-cloud-provider triage`
- **THEN** the analyzer still runs multi-provider aggregation and warns that single-provider selection is deprecated

### Requirement: Local rule path auto-discovery
When local mode requires YARA rules and `-yara-rules` is not explicitly provided, the system MUST resolve rules in this order:
1) `EDR_YARA_RULES`
2) `<exe-dir>/rules/yaraRules`
3) `<exe-dir>/rules`

#### Scenario: Use packaged rules in dist
- **WHEN** the executable is distributed with `rules/yaraRules` under the same `dist` directory
- **THEN** local analysis starts without requiring `-yara-rules`

### Requirement: Distribution packaging contract
Build/distribution artifacts for local analysis SHALL include executable + required YARA-X native library + packaged rules directory under `dist`.
Cloud API key configuration file SHALL remain external and replaceable.

#### Scenario: Distribute without embedded API keys
- **WHEN** an operator receives `dist` artifacts and a separate `edr-cloud.json`
- **THEN** the operator can run local analysis immediately and enable cloud analysis by placing config next to executable
