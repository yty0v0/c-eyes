## Context

本变更为 EDR 增加 Web 站点信息采集能力，目标是统一 Windows 与 Linux 的站点资产模型，支持命令行触发与标准化导出。已有多个扫描能力已采用“信息收集与风险分析解耦”的模式，本能力保持同样边界，只输出资产事实，不输出风险结论。需求中明确要求采集阶段禁止依赖外部命令调用，并要求提示文案中文化、文本编码 UTF-8。

## Goals / Non-Goals

**Goals:**
- 提供 `edr web-site-scan` 命令并支持跨平台采集。
- 支持请求过滤：`groups`、`hostname`、`ip`、`port`、`proto`、`type`、`rootPath`。
- 输出统一字段契约，覆盖站点基础信息、绑定域名、虚拟目录/主目录、协议端口、服务进程及平台特有字段。
- 提供 JSON 与 Excel 两种输出。
- 将内网 IP 与外网 IP 统一为数组字段，完整表达多网卡场景。

**Non-Goals:**
- 不实现风险评分、攻击面评级、漏洞判断或告警结论。
- 不在本变更中引入新的后端存储模型迁移策略（仅定义采集与输出契约）。
- 不扩展为远程主动探测或网络连通性扫描能力。

## Decisions

1. 采集架构采用“平台适配器 + 统一标准模型”。
选择理由：Windows 与 Linux 数据源不同，但上层输出需统一；通过适配器隔离平台差异，避免在业务层分支泛滥。  
备选方案：单一跨平台采集实现。未采用原因：平台差异字段（IIS appPool、Linux permission/acls）会使单实现复杂度过高。

2. 采集约束为“进程内获取”，禁止外部命令行枚举。
选择理由：满足需求约束并降低命令注入与环境依赖风险；同时减少不同发行版命令兼容问题。  
备选方案：优先命令行采集并做兜底。未采用原因：与需求冲突，且稳定性不佳。

3. 输出契约采用“主记录 + 嵌套对象列表”。
选择理由：`domains`、`virtualDir`、`root` 等结构天然具有层次关系，嵌套模型可同时支持 JSON 与 Excel 展平导出。  
备选方案：全部扁平字段。未采用原因：可读性差、字段爆炸、维护成本高。

4. IP 模型采用数组化：`externalIpList`、`internalIpList`，并保留 `displayIp`。
选择理由：覆盖多网卡/多地址场景，符合需求“采集全部内外网 IP”；`displayIp` 作为界面展示兼容字段。  
备选方案：保留单值 `externalIp`/`internalIp`。未采用原因：信息丢失且与需求(3)冲突。

5. CLI 与输出编码规范统一为 UTF-8，提示信息中文。
选择理由：满足用户交互要求，避免多平台终端乱码和导出编码不一致问题。  
备选方案：沿用系统默认编码。未采用原因：跨平台不可预期。

## Risks / Trade-offs

- [不同平台站点类型元数据差异大] → 通过“统一必填核心字段 + 平台可选字段”建模，缺失值用 null/空数组表示。
- [Excel 导出中复杂嵌套字段可读性下降] → 对 `domains`、`virtualDir` 采用 JSON 字符串单元格或多表方案，并在实现阶段固定一种可回归校验的展平策略。
- [无命令行采集可能增加实现复杂度] → 分阶段实现适配器，先覆盖高频站点类型（IIS、nginx），其余类型按统一接口扩展。
- [IP 字段切换为数组影响旧消费方] → 在输出中保留 `displayIp`，并在文档中声明单值 IP 字段废弃。
