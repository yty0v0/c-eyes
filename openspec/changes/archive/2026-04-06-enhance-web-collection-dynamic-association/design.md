## Context

现有 Web 资产采集以静态配置路径为主，适合标准部署，但对自定义启动参数、非默认配置位置和运行态元数据关联不足。为满足更高准确性要求，需要在不破坏现有接口的前提下增加动态关联层。

## Goals / Non-Goals

**Goals:**
- 在静态配置解析后增加动态进程关联层，识别服务类型并提取配置路径。
- 对 `web-site-scan` 补全 `pid`、`cmd`、`user` 等运行态字段。
- 支持非常规配置路径发现并复用既有解析器。
- 保持跨平台与“非外部命令采集”约束。

**Non-Goals:**
- 不引入风险分析、威胁结论或评分。
- 不改变现有 CLI 参数契约与导出格式。

## Decisions

1. 采用“静态优先 + 动态补全”两阶段采集。  
理由：兼顾稳定性与发现率，避免仅依赖动态或仅依赖静态。

2. 动态关联统一基于 `processscan.Scan` 的进程信息。  
理由：复用现有跨平台进程采集能力，避免新增系统依赖。

3. 对动态发现到的配置路径复用现有解析函数。  
理由：保持字段语义一致，降低重复实现与行为分叉。

4. 扫描测试中增加可控开关避免环境波动。  
理由：单测保持确定性，动态关联行为用专门测试验证。

## Risks / Trade-offs

- [进程噪音导致误识别] → 通过服务类型关键词与配置文件后缀双重约束。
- [动态关联影响测试稳定性] → 增加测试开关与独立动态关联单测。
- [不同 Linux 发行版文件元数据差异] → 使用 golden 回归并允许按平台刷新基准。
