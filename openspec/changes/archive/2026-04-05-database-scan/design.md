## Context

现有扫描能力已覆盖端口、进程、计划任务等资产，但缺少数据库维度的统一采集。目标是在 Windows 和 Linux 上以 Go 实现数据库信息收集，不依赖执行系统命令行工具来提取信息，并支持命令行触发扫描与结果导出（JSON/Excel）。该能力仅负责信息采集与结构化输出，不承担风险判定。

## Goals / Non-Goals

**Goals:**
- 提供统一的数据库扫描入口，支持按请求参数过滤目标与结果。
- 建立跨平台采集抽象，支持 Windows/Linux 共享主流程与平台扩展字段。
- 输出统一数据模型，支持 JSON 与 Excel 两种格式。
- 对内外网 IP 使用数组表示，完整保留主机多网卡地址。

**Non-Goals:**
- 不进行漏洞、弱口令、暴露面等风险分析。
- 不实现数据库登录验证、基线核查或修复建议。
- 不引入远程执行命令进行数据采集。

## Decisions

### 1) 采集架构：统一模型 + 平台适配器
- 决策：采用 `collector` 接口，按 OS 实现 `windows` / `linux` 适配器，输出统一 `DatabaseRecord`。
- 原因：跨平台差异主要在发现路径与特有字段；统一模型可简化过滤、序列化与导出。
- 备选方案：按数据库类型拆分完整扫描流程。未采用，因为会重复实现通用过滤与导出逻辑。

### 2) 不通过命令行采集
- 决策：采集层禁止依赖 `exec` 调用外部命令，优先使用系统 API、文件解析与进程信息接口。
- 原因：满足需求约束并减少命令执行权限依赖。
- 备选方案：调用 `ps`/`wmic`/数据库客户端命令补齐字段。未采用，因违反需求边界。

### 3) 查询与过滤
- 决策：请求参数在采集后统一过滤，文本字段支持模糊匹配，数组字段支持包含匹配。
- 原因：先统一归一化再过滤可降低平台分支复杂度。
- 备选方案：边采集边过滤。未采用，会增加适配器实现复杂度。

### 4) 导出策略
- 决策：命令行参数 `-output json|excel` 控制输出格式，默认 `json`；Excel 导出按统一列头映射。
- 原因：满足业务导出需求并保持易用性。
- 备选方案：仅 JSON 输出。未采用，无法满足交付要求。

### 5) IP 字段模型调整
- 决策：将内外网 IP 设计为数组字段（如 `internalIpList`、`externalIpList`），兼容单值映射。
- 原因：主机可能存在多网卡、多地址，单值会丢失信息。
- 备选方案：保留单值字段。未采用，信息不完整。

## Risks / Trade-offs

- [风险] 不同数据库版本配置文件格式差异较大，解析规则易遗漏。  
  -> Mitigation: 按数据库类型维护解析器与回归样本，未知字段保留在扩展区。
- [风险] 不依赖命令行后，部分环境可见性降低。  
  -> Mitigation: 增加多数据源兜底（配置文件、进程参数、服务元数据）并标注字段来源。
- [风险] Excel 列映射随字段扩展变更频繁。  
  -> Mitigation: 维护固定列定义与版本化导出测试。

## Migration Plan

1. 新增 `database-scan` 模块与统一数据结构，不改动现有扫描模块行为。  
2. 接入 CLI 子命令与参数解析，保留默认 JSON 输出保证兼容。  
3. 增加 Excel 导出依赖与格式化测试。  
4. 在 Windows/Linux 环境分别进行样例验证后发布。  
5. 回滚策略：关闭 `database-scan` 子命令入口或回退模块版本，不影响其他扫描能力。

## Open Questions

- Linux HBase 与 Oracle 的 `webAddress` 字段来源优先级是否有既定规范。  
- Windows `dbName` 字段含义是实例名还是默认数据库名，需业务确认。  
- `groups` 过滤是本地映射还是依赖外部资产标签服务，需确认数据来源。
