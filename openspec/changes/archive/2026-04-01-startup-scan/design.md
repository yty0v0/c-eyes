## Context

现有仓库已有多类资产扫描能力，但缺少统一的启动项扫描模块。`docs/startup-scan.md` 要求该能力在 Windows 与 Linux 上可用，且采集过程不得调用外部命令行，仅进行信息收集；同时需要支持查询参数过滤，并提供 JSON/Excel 两种输出格式。设计目标是在复用现有 CLI 与导出框架的前提下，新增可扩展的启动项采集能力并保证字段契约稳定。

## Goals / Non-Goals

**Goals:**
- 提供 `edr startup-scan` 命令，输出结构化启动项信息。
- 支持 `groups`、`hostname`、`ip`、`name`、`initLevel`、`defaultOpen`、`isXinetd`、`showName`、`user`、`enable`、`startType`、`publisher` 过滤参数。
- Linux 与 Windows 均通过进程内系统 API 或系统数据源采集，不依赖外部命令行工具。
- 返回字段键固定、字段名固定；平台不可用字段使用 `null` 或空值兜底，不省略键。
- 复用统一输出链路，支持 JSON/Excel 导出，并与其他扫描能力保持 IP 采集语义一致。

**Non-Goals:**
- 不在本能力内实现风险分析、风险分级、告警或处置动作。
- 不实现远程主机编排扫描，仅处理当前执行节点可访问的数据源。
- 不在本次变更定义新的云侧协议。

## Decisions

1. 新增独立模块 `internal/startupscan`
   - 方案：新增 `types.go`、`scan.go`、`filter.go`、`scan_linux.go`、`scan_windows.go`，并在 CLI 层接入 `startup-scan` 子命令。
   - 原因：与现有扫描能力模块化结构一致，便于测试和维护。
   - 备选：扩展已有扫描模块承载启动项数据。
   - 未选原因：启动项字段和平台差异较大，强耦合会增加后续演进成本。

2. 平台采集实现坚持“无外部命令”约束
   - Linux：通过系统文件/系统接口采集启动项信息，覆盖 init level、默认启用状态、xinetd 标记与 rc0-rc7 状态映射。
   - Windows：通过系统服务相关 API 与系统元数据源采集 `showName`、`user`、`enable`、`startType`、`publisher` 等字段。
   - 原因：满足硬性约束，避免命令输出解析带来的脆弱性。
   - 备选：调用 `systemctl`、`chkconfig`、`sc` 等命令解析文本结果。
   - 未选原因：违反需求约束且跨版本兼容性差。

3. 过滤模型统一收敛到单一参数结构
   - 方案：定义 `StartupScanParams` 承载全部过滤参数，模糊匹配参数采用大小写不敏感子串匹配，数组/布尔参数采用精确匹配。
   - 原因：保证跨平台一致的查询体验，降低 API 认知负担。
   - 备选：按平台拆分参数结构。
   - 未选原因：会导致调用方式不一致，增加使用门槛。

4. 输出模型固定字段并统一空值策略
   - 方案：输出记录固定包含需求定义字段，平台缺失字段使用 `null`（或空集合）兜底，不省略字段键。
   - 原因：减少消费端分支处理，保持输出契约稳定。
   - 备选：按平台裁剪字段。
   - 未选原因：破坏跨平台一致性，影响下游处理逻辑。

5. 导出复用现有 JSON/Excel 管线
   - 方案：复用现有 `-output json|excel` 行为，仅新增启动项字段的 Excel 列映射与序列化适配。
   - 原因：降低重复开发与维护成本，保持命令行为统一。
   - 备选：独立实现启动项导出命令。
   - 未选原因：增加用户学习成本并造成功能分叉。

## Risks / Trade-offs

- [Linux 发行版启动机制差异导致字段完整性不一致] -> 增加数据源优先级与缺省值策略，确保字段键完整返回。
- [Windows 字段（如 publisher）可能受权限或签名信息可用性影响] -> 允许返回 `null` 并记录采集来源，避免中断整体扫描流程。
- [模糊匹配字段较多可能带来过滤性能开销] -> 对高频匹配字段进行预处理（小写缓存）并限制单次扫描输入规模。
- [rc0-rc7 与不同启动管理机制映射存在语义差异] -> 在实现中统一映射规则并通过测试锁定行为。

## Migration Plan

- 第一步：新增 `internal/startupscan` 模块与核心数据结构，定义跨平台统一记录模型。
- 第二步：实现 Linux/Windows 采集器与字段映射逻辑，接入统一过滤流程。
- 第三步：在 CLI 层新增 `startup-scan` 子命令和参数解析，接入 JSON/Excel 输出。
- 第四步：补充单测与跨平台回归测试，验证过滤、字段完整性与导出结果。
- 回滚策略：若出现兼容性问题，移除 `startup-scan` 子命令入口与模块接入，不影响既有扫描能力。

## Open Questions

- `rc7` 字段在原始需求中存在 `rcs` 拼写，是否统一按 `rc7` 作为正式字段名。
- Windows 平台是否需要覆盖“非服务型启动项”（如启动文件夹、Run 注册表）并映射到当前字段模型。
- Excel 导出是否需要支持字段列裁剪或仅保留全字段固定列模式。
