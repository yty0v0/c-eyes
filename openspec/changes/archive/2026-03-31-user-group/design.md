## Context

当前代码库已具备 `accountscan`、`processscan` 等跨平台信息采集能力，但尚无“用户组信息采集”链路。`docs/user-group.md` 明确了请求参数、返回字段、跨平台约束与“仅信息收集不做风险分析”的范围要求，同时要求采集过程不得依赖外部命令执行。本设计需要在延续现有 CLI 使用方式的前提下，新增可独立演进的用户组扫描模块，并保证 Windows/Linux 输出字段稳定一致。

## Goals / Non-Goals

**Goals:**
- 提供 `edr user-group scan` 命令，支持 `groups`、`hostname`、`ip`、`name`、`gid` 过滤。
- 新增 `internal/usergroupscan` 模块，按“平台采集 -> 主机信息补充 -> 过滤 -> 输出规范化 -> 导出”流程实现。
- Linux 通过系统文件解析采集用户组信息；Windows 通过系统账户/组 API 采集用户组信息；全程不依赖外部命令。
- 结果字段覆盖 `docs/user-group.md` 定义，支持 JSON 与 Excel 两种输出格式。
- 平台差异字段采用稳定兜底策略（不可用字段返回 `null` 或空集合），保证消费端兼容。

**Non-Goals:**
- 不新增用户组风险判断、异常评分或告警能力。
- 不实现远程主机用户组拉取与集中化编排。
- 不实现域控/LDAP/AD 深度拉取，仅处理本机可访问用户组信息。
- 不在本次变更中引入新的外部网络依赖。

## Decisions

1. **新增独立模块 `internal/usergroupscan`，并复用现有宿主能力**
   - 方案：新增 `types.go`、`scan.go`、`filter.go`、`scan_linux.go`、`scan_windows.go`，并复用现有主机信息注入能力（`displayIp/externalIp/internalIp/bizGroup*`）。
   - 原因：与现有扫描模块结构一致，降低维护成本并避免跨模块耦合。
   - 备选：直接扩展 `internal/accountscan`。
   - 未选原因：账号与用户组实体语义不同，直接混用会使结构体和过滤逻辑膨胀，长期演进成本更高。

2. **Linux 采用系统文件解析获取用户组**
   - 方案：优先解析 `/etc/group` 获取 `name/gid/members`，必要时结合 `/etc/passwd` 做成员归一化；不调用 `getent` 等外部命令。
   - 原因：满足“禁止命令行采集”约束，且跨发行版行为可控。
   - 备选：通过 CGO 调用 `getgrent` 等 libc 接口。
   - 未选原因：引入 CGO 会增加构建复杂度，与当前纯 Go 交付方式不一致。

3. **Windows 采用 NetAPI 本地组接口采集**
   - 方案：使用 `NetLocalGroupEnum` / `NetLocalGroupGetMembers`（或等价系统 API）采集组名、描述与成员，并映射成员 `name/type`。
   - 原因：接口稳定且可获取组成员类型信息，满足输出字段要求。
   - 备选：通过 WMI 查询本地组与成员。
   - 未选原因：WMI 性能波动较大、查询表达复杂，错误处理和版本兼容性成本更高。

4. **过滤规则采用“主机字段模糊匹配 + 组字段精确/模糊组合”**
   - 方案：`hostname/ip/name` 使用大小写不敏感子串匹配；`groups` 命中任一业务组 ID；`gid` 为 Linux 精确匹配。
   - 原因：与现有扫描体验一致，满足文档中的模糊查询要求。
   - 备选：所有字段统一精确匹配。
   - 未选原因：会显著降低检索可用性，不符合既有 CLI 使用习惯。

5. **输出契约固定，字段不可缺省**
   - 方案：顶层输出固定 `total + rows`；`rows` 中字段键固定存在，平台不支持字段以 `null` 或空集合返回。
   - 原因：提升下游消费稳定性，避免按平台分支解析。
   - 备选：按平台裁剪字段，未采集字段直接省略。
   - 未选原因：会增加消费端分支复杂度，并导致接口契约不稳定。

6. **Excel 导出复用统一导出管线**
   - 方案：在现有 CLI 输出框架上新增用户组导出表头和字段映射，保持 `-output json|excel` 与 `-excel` 参数行为一致。
   - 原因：减少重复实现并维持一致的命令使用体验。
   - 备选：为用户组扫描单独实现一套导出命令。
   - 未选原因：会造成参数和行为不一致，增加文档与维护负担。

## Risks / Trade-offs

- [Linux 组成员来源可能不完整（如 NSS/外部目录服务）] -> 明确本次范围仅保证本机可访问系统文件信息，并在文档注明边界。
- [Windows API 在不同版本上字段可用性差异] -> 采用“可用即填充，不可用返回 null”的统一兜底策略。
- [成员列表可能很大导致输出体积上涨] -> 提供可配置上限或分页策略作为后续扩展点，本次先保证完整输出。
- [业务组 ID 与主机本地组之间语义映射可能不一致] -> 将 `groups` 过滤限定为主机元数据维度，用户组实体仍按 `name/gid` 匹配。
- [Excel 导出字段多时易出现列顺序漂移] -> 固化表头顺序并加导出快照测试。

## Migration Plan

- 第一步：新增 `internal/usergroupscan` 模块骨架与类型定义，接入 CLI 子命令入口。
- 第二步：实现 Linux/Windows 采集逻辑、过滤逻辑与统一结果构造。
- 第三步：接入 JSON/Excel 输出与命令参数校验，补齐单元测试与示例文档。
- 回滚策略：若出现兼容问题，回退本次变更并移除 `user-group` 子命令，不影响现有 `process/file/account/risk` 能力。

## Open Questions

- `groups` 请求参数是否始终代表“业务组 ID”（而非本机用户组 ID）并由主机元数据匹配。
- Windows 成员 `type` 的枚举是否需要与既有账号扫描枚举完全复用，或在文档中单独定义。
- Excel 导出是否需要支持自定义列选择，还是固定输出全部字段即可。
