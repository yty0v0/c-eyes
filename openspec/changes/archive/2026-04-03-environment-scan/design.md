## Context

本变更新增环境变量信息采集能力，目标是在 Linux 与 Windows 上提供统一的查询与输出契约，并纳入现有 EDR CLI 体系。当前系统已有账号、计划任务等资产采集能力，但缺少环境变量采集，导致排障与资产审计链路不完整。该能力必须满足以下约束：仅进行信息收集、不做风险判断；采集实现不得依赖外部命令行工具；输出需支持 JSON 与 Excel。

## Goals / Non-Goals

**Goals:**
- 提供 `edr environment-scan` 命令并支持过滤参数：`groups`、`hostname`、`ip`、`key`、`value`、`user`、`sysEnv`。
- 建立跨平台统一输出结构，包含主机元信息与环境变量字段。
- 将主机内外网 IP 统一为列表字段：`externalIpList`、`internalIpList`。
- 支持 JSON 与 Excel 两种输出格式。
- 明确能力边界仅为信息收集与标准化输出。

**Non-Goals:**
- 不在本变更中实现风险评估、威胁判定、告警打分。
- 不在本变更中扩展远程代理分发、批量并发调度策略。
- 不在本变更中定义新的持久化模型或历史版本管理能力。

## Decisions

1. 采集方式采用“进程内平台适配器”模式  
   - 决策：在 Linux 与 Windows 分别实现采集适配层，通过系统 API / 系统数据源读取环境变量，不调用外部命令。  
   - 原因：满足“禁止命令行采集”的强约束，同时便于控制权限边界与错误处理。  
   - 备选方案：通过 shell 命令（如 `env`、`set`）采集。  
   - 放弃原因：不符合约束，且输出稳定性与安全边界较弱。

2. 数据模型采用“记录级标准化 + 字段可空”  
   - 决策：每条记录保持固定字段集，平台差异字段不可得时返回 `null` 或空数组，不省略键。  
   - 原因：便于上层消费与导出，减少解析分支。  
   - 备选方案：按平台返回不同字段结构。  
   - 放弃原因：会增加调用方兼容成本。

3. IP 字段统一采用列表表达  
   - 决策：输出 `externalIpList` 和 `internalIpList`，并保留 `displayIp` 作为主展示 IP。  
   - 原因：同一主机多网卡场景下更准确，与现有扫描能力对齐。  
   - 备选方案：仅保留单值 `externalIp`/`internalIp`。  
   - 放弃原因：信息丢失，无法表达全量地址。

4. 输出格式通过统一序列化层切换  
   - 决策：先构建统一内存结果，再按参数选择 JSON 或 Excel 编码器。  
   - 原因：减少重复逻辑，确保两种格式字段一致。  
   - 备选方案：采集阶段分叉生成 JSON/Excel。  
   - 放弃原因：耦合高，测试复杂度更高。

## Risks / Trade-offs

- [Risk] 跨平台权限差异导致部分变量不可见 -> Mitigation: 对不可访问字段返回空值并记录结构化错误上下文，不中断整体返回。  
- [Risk] 大量环境变量导致导出文件体积较大 -> Mitigation: 在查询层优先应用过滤条件，并在 Excel 导出时采用流式写入。  
- [Risk] `sysEnv` 语义在不同平台映射不一致 -> Mitigation: 在规范中固定判定规则，并在实现中集中封装映射函数。  
- [Risk] 模糊匹配规则差异引发结果不一致 -> Mitigation: 统一采用大小写不敏感子串匹配并补充跨平台测试样例。
